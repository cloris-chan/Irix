## Current Design Notes

- The Counter PoC button currently applies press `TranslateY` to the same visual node that carries action/hit-test metadata. This intentionally exercises active/presented hit-test remapping: pointer input can target the visual position while a transform animation is in flight or has just completed. Single-target commit preserves the completed presented transform for hit testing instead of clearing it through the cancel path, so a downward press motion should not flicker when the pointer leaves or re-enters across the lower edge after completion. Flicker at the upper edge can still be a valid consequence of moving through the visual displacement band because canceling the downward offset can put the pointer back inside the original bounds. Future public controls should keep the outer interaction, capture, focus, and release-activation bounds stable, and apply hover/press motion to an inner visual/content layer so visual feedback does not move the control's own activation boundary.

Performance allocation note: `layout.result=0` means the retained layout result shell is no longer a heap object, `pipeline.snapshot.retainedInput=0` means the retained input snapshot shell is no longer a heap object, `retainedFrame=0` in warm scroll/scale means retained hit-target publication is not defensively copied into the retained frame, `pipeline.classify=0` means empty/single dirty classifications are not published through arrays, `layout.elementsArray=0`, `layout.treeNodesArray=0`, `layout.dirtyRanges=0`, and `record.dirtyRanges=0` mean common layout and dirty-range publications stay in `LayoutElementList`, `LayoutTreeNodeList`, `LayoutElementRangeList`, and `IndexRangeList`, `layout.scrollDiagnosticsArray=0` means common scroll diagnostics stay in `ScrollContainerDiagList`, and `hitTargets=0` means common clean and dirty hit-target publications stay in `HitTargetList`. `drawRecord=0 B/frame` in warm scroll/scale means recorder-owned typed command owners retain command backing capacity behind generation-checked batch views; resource/style publication and small command-build accounting still need reserved-capacity ownership. `layout.nodeWalk=0` means layout walking itself is not allocating. `DrawCommandRecordResult`, `RenderPipelineRetainedInputSnapshot`, `LayoutDirtyClassificationList`, `IndexRangeList`, common `ElementCommandRangeList`, `ScrollContainerDiagList`, `HitTargetList`, `LayoutElementList`, `LayoutTreeNodeList`, `LayoutElementRangeList`, and typed command owner generation tokens are readonly/value publications now; result/snapshot shells, common dirty-classification/range arrays, small element-command range arrays, common scroll-diagnostic arrays, common hit-target arrays, common small layout arrays, recorder command-owner shells, and warm draw-command backing arrays are not retained per-frame identities. Large layout publications, style-only patched element copies, retained snapshot referenced arrays/lists, multi-classification dirty arrays, non-inline hit-target arrays, non-inline scroll diagnostic arrays, large draw-command element range arrays, and frame resources are retained across frames and must not expose mutable pooled storage without generation checks. The no-allocation target does not allow generic array pooling as a shortcut; it requires typed capacity-owned publication pages/slabs with generation fences so retained readers see immutable views while writers advance a new generation. The first control composition scratch slices now avoid production button bundle/split temporary arrays, publish compact control metadata through `VirtualNodePropertyList`, freeze remaining child publication through `VirtualNodeChildList`, and publish the Counter root child array without staging scroll rows in a temporary array while preserving the semantic style bridge. The current `tree.buildRoot.button.childrenList` hotspot in `--diagnose-text-cache` remains the expected retained-publication cost of lowering controls into the simplified Container/Content IR, while control metadata now reports `tree.buildRoot.button.propertyList=0`; optimize child-list backing storage through publication-slab ownership next, not by reintroducing button nodes or z-index/render-style state. `--diagnose-render-steady-state-allocation 30` is the inner core-pipeline baseline; a later reserved-capacity guard must expand the scope to control/tree publication, resource/style publication, backend resources, and cache growth before any full zero-allocation claim. Reopen retained-array, snapshot-reference, or tree-builder work through [Render-Pipeline-Performance-Architecture.md](Render-Pipeline-Performance-Architecture.md).

## Active Source Guards

Keep guards green for these non-negotiable boundaries:

- Final text composition stays on the D3D12 GlyphAtlas path; DirectWrite/WIC remain source-data providers.
- No runtime shader compile in active renderer source.
- No raw retained text string path in framework/core rendering or layout state.
- No public string style/value factory path, no public renderer brush/material/radial-gradient/image authoring path, no public UI authoring split by layout/visual/composition processing layers, no public transition scheduler/theme/cascade surface, no legacy attribute-era API, public `VirtualNodeProperty` construction, silent `PropertyValue` getters, ref-struct leakage into retained/batch state, or framework/backend `record struct` outside the existing PoC authoring exception.
- No treating current `DrawColor` / `WindowColor` SDR payloads as the canonical Irix color value; canonical color work must preserve the [Color-Pipeline.md](Color-Pipeline.md) boundary.
- No internal material kind beyond `None`, `SolidColor`, and `LinearGradient`; public non-solid authoring is limited to semantic two-stop directional background paint and typed border stroke until a dedicated slice expands the contract.
- No HDR/scRGB/Rec.2100 output mapper, D3D12 HDR swapchain format, D3D12 color-space selection, system SDR brightness, or tone-mapping implementation before the guarded HDR output mapping slice exists.
- No per-monitor, backend-present, vblank, fence, or refresh-rate animation clock; those signals are pacing/diagnostics only and all animation progress must flow through the Irix composition clock / `CompositionTimestamp` boundary.
- No compositor/runtime code extraction before the relevant style, animation, composition, scroll, or input/control ownership contract is written.
- No viewport-space internal offscreen/render-target cache on the active composition path.
